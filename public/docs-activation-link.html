Activation link integration

When a buyer pays for your product through Boost Boss, we email them a one-click "Activate your account" button. The button links to your signup page with a single-use code appended as ?bb_token=<code>. Your signup endpoint validates that code with our API and creates the account already on the paid plan — no per-order action from you, no separate redemption page for the buyer.

The short version.

Read bb_token from the URL. POST it to /v1/vouchers/validate with your seller secret. If valid: true, create the paid account and POST to /v1/vouchers/redeem to mark it consumed. That's the entire integration — usually 30–60 lines on your signup endpoint.

1. The flow at a glance

1
Buyer pays
PayPal capture succeeds on Boost Boss's hosted checkout.
2
BB emails buyer
Branded confirmation with an "Activate your account" CTA. We do this automatically — you do nothing.
3
Buyer clicks the CTA
They land on your signup page at your-url?bb_token=BB-XXXX-XXXX-XXXX.
4
Your code validates + creates
Call validate, create the paid account, call redeem. Log them in.

2. One-time setup

In your product modal in the advertiser dashboard, on the Fulfillment tab, set two fields:

  • Activation page URL. Where buyers land — your signup or register page (e.g. https://yourapp.com/signup). BB appends ?bb_token=<code> to this URL.
  • Webhook signing secret. Any random string ≥ 32 chars. This doubles as your seller secret for the voucher API. Store it in your backend environment as BB_SELLER_SECRET.

Both fields lock once your product is approved. If you mis-configure either, you'll need to duplicate the product as a new draft and re-submit for audit — so triple-check before submitting.

3. The two endpoints you'll call

POST /v1/vouchers/validate

Read-only check. Returns the voucher state. Safe to call as many times as you want — no side effects. Use this on form load (to show "you're activating Pro" pre-fill the email field) or on form submit before you create the account.

POST https://boostboss.ai/v1/vouchers/validate
Content-Type: application/json

{
  "code":          "BB-A8X3-92K4-7QM2",
  "seller_secret": "your-32-plus-char-secret-from-the-fulfillment-tab"
}

Response when the voucher is good to use:

{
  "success": true,
  "valid":   true,
  "voucher": {
    "code":        "BB-A8X3-92K4-7QM2",
    "status":      "issued",
    "expires_at":  "2026-09-12T00:00:00Z",
    "expired":     false,
    "product":     { "id": "...", "name": "Pro Plan", "sku_type": "subscription", "package_duration_days": 365 },
    "buyer":       { "email": "buyer@example.com" },
    "transaction_id": "tx_abc123"
  }
}

The valid boolean is the one to gate on. It is true only when status === "issued" (i.e. paid, not yet redeemed, not refunded, not expired). The buyer.email field is supplied so you can pre-fill the signup form or correlate to existing accounts; you do not need to enforce that the signup email matches.

POST /v1/vouchers/redeem

State-changing. Flips the voucher from issued → redeemed and stamps the time. Idempotent — re-submitting the same code returns deduped: true without flipping anything, so it's safe to call from a webhook handler that might fire twice.

POST https://boostboss.ai/v1/vouchers/redeem
Content-Type: application/json

{
  "code":          "BB-A8X3-92K4-7QM2",
  "seller_secret": "your-32-plus-char-secret",
  "buyer_metadata": { "account_id": "u_new_42", "plan": "pro" }
}

Response on success:

{ "success": true, "redeemed": true, "voucher": { ... } }

Already redeemed (idempotent re-submission):

{ "success": true, "redeemed": true, "deduped": true, "voucher": { ... } }

Refunded / expired / revoked (HTTP 409):

{ "success": false, "redeemed": false, "reason": "refunded" | "expired" | "revoked", "voucher": { ... } }

4. Code samples

Drop-in handlers for a typical signup endpoint. Adapt to your framework — these show the shape, not the framework wiring.

// POST /signup  (Express / Fastify / any Node framework)
async function handleSignup(req, res) {
  const { email, password, bb_token } = req.body;

  if (!bb_token) {
    return createFreeAccount(req, res); // fall through to your normal flow
  }

  // 1. Validate the voucher with BB
  const v = await fetch('https://boostboss.ai/v1/vouchers/validate', {
    method:  'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      code:          bb_token,
      seller_secret: process.env.BB_SELLER_SECRET,
    }),
  }).then((r) => r.json());

  if (!v.success || !v.valid) {
    return res.status(400).json({ error: 'Invalid or expired activation token' });
  }

  // 2. Create the paid account on the plan attached to this product
  const account = await db.users.create({
    email,
    password_hash: await hash(password),
    plan:          'pro',
    paid_via:      'boostboss',
    bb_transaction_id: v.voucher.transaction_id,
  });

  // 3. Mark the voucher consumed (idempotent — safe to retry)
  await fetch('https://boostboss.ai/v1/vouchers/redeem', {
    method:  'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      code:           bb_token,
      seller_secret:  process.env.BB_SELLER_SECRET,
      buyer_metadata: { account_id: account.id, plan: 'pro' },
    }),
  });

  // 4. Log them in
  return loginUser(req, res, account);
}
# POST /signup  (Flask / FastAPI / Django — same shape)
import os
import requests

BB_BASE = 'https://boostboss.ai'

def handle_signup(req):
    email     = req.form['email']
    password  = req.form['password']
    bb_token  = req.args.get('bb_token') or req.form.get('bb_token')

    if not bb_token:
        return create_free_account(req)  # your existing free-tier flow

    # 1. Validate the voucher with BB
    v = requests.post(
        f'{BB_BASE}/v1/vouchers/validate',
        json={
            'code':          bb_token,
            'seller_secret': os.environ['BB_SELLER_SECRET'],
        },
        timeout=10,
    ).json()

    if not v.get('success') or not v.get('valid'):
        return {'error': 'Invalid or expired activation token'}, 400

    # 2. Create the paid account
    account = db.users.create(
        email=email,
        password_hash=hash_password(password),
        plan='pro',
        paid_via='boostboss',
        bb_transaction_id=v['voucher']['transaction_id'],
    )

    # 3. Mark the voucher consumed (idempotent)
    requests.post(
        f'{BB_BASE}/v1/vouchers/redeem',
        json={
            'code':           bb_token,
            'seller_secret':  os.environ['BB_SELLER_SECRET'],
            'buyer_metadata': {'account_id': account.id, 'plan': 'pro'},
        },
        timeout=10,
    )

    # 4. Log them in
    return login_user(account)

5. No backend? Use the Zapier path

If you don't have a backend that can call APIs — typical for Webflow, Carrd, Shopify-without-custom-apps, Bubble lite plans — you can integrate using the fulfillment webhook instead of the voucher API.

The fulfillment webhook is configured one tab over from the activation URL, in the same product modal. BB POSTs a signed payload to whatever URL you set every time a payment captures, before the buyer even clicks the email CTA. Point that URL at a Zapier (or Make, or n8n) webhook trigger and let the no-code automation do the work.

Recommended Zapier recipe

  1. Trigger: Webhook → Catch Hook. Copy the catch URL into your product's "Fulfillment webhook URL" field.
  2. Filter (optional): Only continue if event == "purchase.captured".
  3. Action: Create user in your CRM / mailer / app DB. Map buyer_email → email, transaction_id → custom field, product.name → plan tag.
  4. Action: Email the buyer their custom welcome / setup instructions (in addition to BB's confirmation email).

With this path you don't call the voucher API at all — the webhook IS the activation signal. Buyers who land on your activation URL won't be auto-validated, so configure your activation URL to point at a static "Thanks! Check your email for setup instructions" page, since onboarding happens by email rather than by signup form.

Want signature verification?

The webhook payload is signed with the same fulfillment_webhook_secret. Zapier can verify the X-BB-Signature header in a Filter step if you want to be extra safe — instructions are in the API reference.

6. Security model — what makes the link safe

The activation link is a single-use bearer credential. Anyone who holds the URL can redeem the paid account once. This is the same model as Steam keys, AppSumo codes, Gumroad license keys, and most "you paid, here's your access" flows. Buyers can sign up under a different email than they paid with — useful when someone pays with personal email and registers under work email, or vice versa. That flexibility is intentional.

Five primitives keep this safe:

PrimitiveWhat it stopsStatus
Single-use redemption An already-redeemed code can't claim a second account. Re-submission returns deduped. enforced server-side
Unguessable code 12 random characters (~32^12 ≈ 1.15 × 1018 possibilities). Effectively impossible to brute-force. enforced server-side
Time-bound Each voucher has an expires_at. After that, status auto-flips to expired. enforced server-side
Seller secret required to redeem A stranger who sees a URL can't call BB's API themselves — only your endpoint has the secret. Constant-time comparison prevents timing attacks. enforced server-side
TLS in transit Email → activation URL → seller signup → BB API. No plaintext hops. enforced

What this doesn't stop: if the buyer forwards their email to a friend before redeeming, the friend can claim the account. We treat that as the same risk as forwarding any other digital purchase receipt — the buyer is responsible for keeping their access credentials private. If your product has compliance constraints that require seat-bound assignment (B2B / enterprise SKU where the paying email must equal the using email), email support@boostboss.ai — we can enable a per-product email-match flag.

7. Testing your integration

Before shipping to real buyers, make a $1 test purchase of your own product:

  1. Set your product's lowest pricing plan to $1 (you can revert after testing).
  2. Buy it from a different browser session using a real email you control.
  3. Wait for the activation email, click the CTA. Confirm your signup endpoint receives bb_token as a query parameter.
  4. Walk through your signup form. Confirm validate returns valid: true and that the account is created on the paid plan.
  5. Try the link again from a fresh incognito window — confirm it shows the appropriate "already redeemed" error in your UI.

BB takes its commission split out of the $1 too, so you'll see a small earnings line in your Payouts page that you can also use to test the end-to-end money loop.

8. Troubleshooting

401 bad_secret — your seller_secret doesn't match the fulfillment_webhook_secret you set on the product. Re-copy it from the product modal and re-deploy.

401 no_secret — you never set a webhook signing secret on the product. Open the product modal, Fulfillment tab, fill the field, save, re-submit for audit if needed.

404 voucher_not_found — the code is malformed or doesn't exist. Likely a typo or a code from a different environment.

409 reason: refunded — the buyer was refunded after redeeming. Don't grant access. Show them a "this purchase was refunded — please contact support" message.

409 reason: expired — the voucher's redemption window passed before they clicked. By default this is 90 days; you can change the window on the product. Direct them to support@boostboss.ai.

The webhook fires but the activation link doesn't work — your activation URL field on the product is wrong (typo, redirects, blocks query strings). Confirm by manually appending ?bb_token=test to it in a browser. If the page errors, fix it on your side.
