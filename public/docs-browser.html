Lumi for Browser App

One script tag. 8 placements.

The fastest install in adtech. Paste one <script> in your <head> and Boost Boss handles everything else — placement auto-detection, ad fetching, impression tracking, click attribution.

Install

Paste this into your site's <head>. Replace YOUR_PUBLISHER_ID with the ID shown on your publisher dashboard:

<script async src="https://boostboss.ai/lumi/v1.js#YOUR_PUBLISHER_ID"></script>

Refresh your page. The Browser App verify badge on your dashboard flips to Connected within 30 seconds.

That's the whole install. Yes, really.

What this gives you

All 8 Browser App placements come live with the install. The Lumi runtime auto-mounts every placement — no slot tags, no flags, no opt-in step. Boost Boss's auction picks which placement serves per ad request based on user intent and real-time fill.

  • Corner unit ✅ auto-mounted — sticky dismissable card pinned bottom-right
  • Settings page slot ✅ auto-mounted — sponsored card on routes matching /settings · /billing · /account · /profile
  • Page interstitial ✅ auto-mounted — full-page sponsor on SPA route changes (once per page, 60s grace after install)
  • Loading-state ad ✅ auto-mounted — auto-detects aria-busy / .spinner / .loading visible >2s
  • Sponsored citation ✅ auto-mounted — heuristic-detected AI-response container, falls back to a safe default
  • Suggested chip ✅ auto-mounted — heuristic-detected suggested-action row, falls back to a safe default
  • Inline sponsored card ✅ auto-mounted — heuristic-detected feed container, falls back to a safe default
  • Empty-state hero ✅ auto-mounted — heuristic-detected empty view, falls back to a safe default

v1.5 ships full auto-mount for all 8 placements. Lumi detects common UI patterns (chat containers, suggestion bars, feed lists, empty states, loading spinners) and falls back to a safe default location when an ideal placement can't be confidently detected. Auto-mount never injects into code blocks, password fields, system messages, or navigation chrome — see the safety rules below.

Optional — pin a placement to an explicit spot

If you'd rather pick the exact location for a placement instead of relying on heuristic detection, drop a slot marker. It overrides auto-detection for that placement only:

<!-- After your AI response — Lumi will mount a sponsored citation HERE -->
<div data-lumi-slot="citation"></div>

<!-- In your suggested-action row -->
<div data-lumi-slot="chip"></div>

<!-- Between content blocks in your feed -->
<div data-lumi-slot="card"></div>

<!-- Inside an empty view -->
<div data-lumi-slot="hero"></div>

<!-- During a loading state -->
<div data-lumi-slot="loading"></div>

Slot markers are optional. Lumi re-scans for new data-lumi-slot markers every 2 seconds for up to 20 seconds after install, so React/Vue components that mount asynchronously are caught automatically. Already-filled slots are remembered and not re-rendered.

Suppress a specific placement

Auto-mount is on by default per Publisher Agreement §4.1. To suppress a specific placement, add a data-lumi-disable marker anywhere in the DOM:

<meta data-lumi-disable="corner" />       <!-- no corner unit -->
<meta data-lumi-disable="interstitial" /> <!-- no page interstitial -->
<meta data-lumi-disable="citation" />     <!-- no sponsored citation -->
<meta data-lumi-disable="loading" />      <!-- no loading-state ad -->
<meta data-lumi-disable="all" />          <!-- nothing renders -->

Suppressing placements reduces your impression volume and earnings. Suppression is for genuine UX or compliance reasons, not for limiting Lumi's reach.

What auto-mount never touches

The runtime refuses to inject into structural risky containers: code blocks (pre / code / .monaco-editor / .CodeMirror), form inputs (input / textarea / select / [contenteditable]), system messages ([role="alert"] / [role="status"]), and site chrome (header / nav / [role="banner"] / [role="navigation"]). If a placement causes a real UX issue, email support@boostboss.ai and we'll relocate it.

Strict Content-Security-Policy

If your site enforces a strict CSP header, add boostboss.ai to two directives:

Content-Security-Policy:
  script-src 'self' https://boostboss.ai;
  connect-src 'self' https://boostboss.ai;

No other directives need to change. Lumi does not use eval, inline scripts, or remote-loaded code.

Privacy posture

Active tab only · no persistence · no cross-site tracking.

Lumi reads only what's visible on the active tab where your site is loaded. Nothing is persisted to cookies, localStorage, sessionStorage, or IndexedDB. The only data that leaves the page is: publisher ID, page URL host + path, a per-page-load ephemeral session UUID, and a short context summary for ad relevance matching.

Lumi never reads from password fields, credit-card fields, or any input with autocomplete="cc-*". Full posture documented in our privacy policy.

Verifying your install

After pasting the script tag and reloading your site, three things should happen:

  1. Your dashboard's Browser App verify badge flips to Connected within 30 seconds.
  2. Your browser's developer console shows: ▲ Lumi · Browser App · pub_... · Connected
  3. The first real ad impression starts serving within 60 seconds of your first user session.

If you don't see the verify badge update, open the browser console — Lumi logs why it didn't connect (missing publisher ID, blocked by CSP, blocked by an ad blocker, etc.).

Uninstall

Delete the <script> tag from your <head>. That's it. No cleanup. No leftover storage to clear. The runtime never wrote anywhere outside the active page in memory, and that's gone the moment the user navigates away.

FAQ

Will ad blockers block this?

Some will. Lumi loads from boostboss.ai; if a user runs uBlock Origin and we're on their blocklist, the script won't execute. That's the same constraint every external script faces (Stripe, Intercom, GA4 included). We are not a tracking pixel — we serve contextual ads — so we are not on most default blocklists.

Does it slow down my page?

No. The script is loaded async, doesn't block first paint, and the runtime defers all work to after DOMContentLoaded. Network footprint: ~30KB on first hit, cached for 24h after that.

What if I want to opt a specific page out?

Add data-lumi-disable to any element. Lumi skips that subtree. Coming in v1 with the placement renderers.

I'm on Lovable / Bolt.new / Webflow / Framer. Can I install?

Yes. All those platforms have a "custom head" or "embed script" field — paste the script tag there.
