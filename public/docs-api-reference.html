Boost Boss API reference

Every shipped endpoint, organized by role. Real paths, real auth, real response shapes pulled from the live handlers.

★ Start hereUpdated Jun 2026·4 min read
Docs / API reference

Every shipped endpoint, organized by role. Real paths, real auth, real response shapes pulled from the live handlers.

Boost Boss exposes a single REST surface at https://boostboss.ai covering four roles: publishers (monetizing AI tools), advertisers (boosting their products), sellers (using the MoR storefront), and affiliates (earning commission on share links). The same account model unifies all four — you sign up once, opt into whatever role(s) you need. A machine-readable spec is at /openapi.json.

Jump to the role you came here for:

PubPublisher endpoints

MCP JSON-RPC auction, REST ad-request, event tracking beacons.

AdvAdvertiser endpoints

Campaigns CRUD, stats, deposits, conversion postback.

SelSeller (MoR) endpoints

Products, pricing plans, orders, vouchers, payouts.

AffAffiliate endpoints

Share-link redirect, sales report, commission state.
!
Spec, not snapshot. Most endpoints described here are live; a few are marked roadmap. When in doubt, the source of truth is the JS handler in api/<name>.js on GitHub and the OpenAPI spec at /openapi.json. Benna is currently a deterministic targeting-overlap scorer, not a learned model — see Benna for the honest status.

Authentication

Boost Boss uses JWT bearer tokens for the authenticated REST endpoints. There are no Stripe-style sk_live_* opaque keys — sign in with email + password, get back a signed JWT, and pass it on every subsequent request.

Login

POST/api/auth?action=loginExchange credentials for a JWT.
cURLCopy
curl https://boostboss.ai/api/auth?action=login \
  -H "Content-Type: application/json" \
  -d '{ "email": "you@example.com", "password": "..." }'
JSON · 200 OKCopy
{
  "session": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "expires_in":   2592000,
    "token_type":   "Bearer"
  },
  "user": { "id": "...", "email": "you@example.com", "role": "advertiser" }
}

The token is a standard signed JWT — you can decode it client-side to read your user ID and role without making a network call. Pass it back on every subsequent request:

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

Two special cases

Two surfaces don't use the user JWT. They use shared secrets you configure once per resource:

Seller voucher API (seller_secret)

The voucher endpoints (validate / redeem) authenticate via a per-product secret passed in the request body as seller_secret. The secret is the fulfillment_webhook_secret field you set on your product in the advertiser dashboard. Compared with crypto.timingSafeEqual server-side; mismatches return 401 bad_secret.

OpenRTB BBX seat key

DSPs hitting /api/rtb use a seat key issued at integration time, sent as a normal bearer header: Authorization: Bearer bb_seat_<key>. Contact support@boostboss.ai for seat onboarding.

i
Storage gotcha. The advertiser dashboard stores the token in localStorage.bb_session_token as a JSON-encoded object — not the raw token string. Production frontends must JSON.parse and pull .token before sending. Sending the raw blob causes silent gate-fallthrough on owner-only routes.

Common patterns

Request and response shape

All endpoints accept and return JSON with Content-Type: application/json. The only exceptions are /api/track (also accepts a 1×1 GIF beacon GET) and /s/<token> (302 redirect).

Action multiplexing

Many endpoints multiplex multiple operations through one URL using an action query param — e.g. /api/campaigns?action=create, /api/auth?action=login. This is a deliberate choice to keep the Vercel function count low; treat each action value as a distinct logical endpoint.

Error envelope

Every error returns a JSON body with a machine-readable error field and (where applicable) a code field for programmatic branching. See Error codes.

{ "error": "Insufficient ad credit. Available: $34.30, requested: $50.00.",
  "code":  "insufficient_credit",
  "available": 34.30,
  "requested": 50.00 }

Idempotency

Where it matters — voucher redemption, conversion postback, PayPal capture — endpoints are idempotent. Re-submitting the same operation returns the existing record with a deduped: true flag rather than producing duplicates. The conversion postback also accepts an idempotency_key field on the request body.

PubPublisher endpoints

Publishers monetize AI surfaces — MCP servers, AI apps, browser extensions, Discord/Slack/Telegram bots. Pick the integration matching your deployment surface; all three call the same underlying auction.

  • MCP servers → talk to /api/mcp directly via JSON-RPC, or install @boostbossai/lumi-mcp.
  • Browser-side AI apps → drop in the /lumi.js script tag or install @boostbossai/lumi-sdk.
  • Server-side bots → POST to /v1/ad-request with a publisher Bearer key.

Publisher revenue settles per your signed Publisher Agreement. See Payouts for cycle and provider details.

MCP JSON-RPC endpoint

POST/api/mcpModel Context Protocol — JSON-RPC 2.0.

The endpoint the @boostbossai/lumi-mcp SDK talks to. Implements three MCP methods: initialize (handshake), tools/list (enumerate the two tools), and tools/call (invoke get_sponsored_content or track_event). Auth: Authorization: Bearer <publisher_api_key>.

Request — tools/call · get_sponsored_content

JSON-RPCCopy
{
  "jsonrpc": "2.0",
  "id":      1,
  "method":  "tools/call",
  "params": {
    "name":      "get_sponsored_content",
    "arguments": {
      "context":      "refactor this python function to use async",
      "active_tools": ["filesystem", "github"],
      "host_app":     "cursor",
      "surface":      "chat"
    }
  }
}

Response

A Benna-scored first-price auction result wrapped in the standard JSON-RPC envelope. result.content[0].text is a JSON-encoded payload with the ad fields plus signed tracking URLs. Beacons are pre-computed; track_event is provided as a convenience when the SDK can't fire URLs directly.

AdvAdvertiser endpoints

Advertisers boost their products through the SuperBoost direct path. The Pilot Console UI talks to /api/products (boost_* fields) and /api/pilot-spend. The /api/campaigns endpoints documented below remain functional for Advanced mode. Auth: JWT bearer obtained from /api/auth?action=login as role advertiser.

List campaigns

GET/api/campaigns?advertiser_id=<uuid>All campaigns for the authenticated advertiser.

Returns the array under campaigns. Filters: status (in_review / active / paused / rejected), limit, offset. Use ?id=<uuid> to fetch a single campaign.

Update campaign

PATCH/api/campaigns?action=updateModify fields on an existing campaign.

Pass id in the body. Any field from create can be updated. Editing creative re-runs policy. Setting status: "paused" halts spend immediately.

Get stats

GET/api/stats?type=advertiser&id=<uuid>Aggregate spend, impressions, clicks, conversions.

Returns daily series + totals over the time range. Break down by campaign, format, host app, integration method, or day. The publisher-side counterpart is ?type=developer&key=<publisher_id>.

Deposit funds

POST/api/billing?action=create_paypal_orderStart a PayPal order to add advertiser balance.
POST/api/billing?action=capture_paypal_orderCapture the order after the buyer approves on PayPal.
GET/api/billing?action=balance&id=<uuid>Current advertiser balance.
GET/api/billing?action=history&id=<uuid>Paginated transaction history.

Advertiser deposits run through PayPal end-to-end, matching publisher and seller payouts. The single-provider rule (PayPal pay-in + payout, both rails) ships from a Taiwan corporate entity, where local law requires pay-in and payout to flow through the same provider. See Payouts for the symmetric payout side.

Flow

  1. Frontend POSTs create_paypal_order with the deposit amount. Backend creates a PayPal order and returns an approval_url.
  2. Frontend redirects the advertiser to approval_url. PayPal returns them to your configured return URL with the order ID echoed back.
  3. Return URL handler POSTs capture_paypal_order with the order ID. Backend calls PayPal Capture, credits the advertiser balance, and returns the new balance.
  4. The PayPal webhook reconciles asynchronously so dropped captures self-heal.
i
Legacy Stripe fallback. The codebase still contains a Stripe deposit path behind PAYIN_PROVIDER=stripe for development convenience. It will be removed once the Singapore corp move opens up multi-provider options. Don't integrate against it in production.

SelSeller endpoints (Merchant of Record)

Sellers register products in BB, BB hosts the checkout, splits the proceeds per Commission model v2, and emails buyers an activation link. The seller's job is to configure their product once and accept an inbound webhook (or call the voucher API on signup) when a buyer activates.

For the full activation-link integration walk-through with code samples in Node / Python / PHP and a Zapier no-code path, see Activation link integration. This section is the API reference; that page is the integration guide.

Validate voucher

POST/v1/vouchers/validateRead-only check before granting access.

Call from your signup endpoint when a buyer lands with ?bb_token=<code>. Returns the voucher state + product context + buyer email. The valid boolean is true only when status is issued.

cURLCopy
curl https://boostboss.ai/v1/vouchers/validate \
  -H "Content-Type: application/json" \
  -d '{
    "code":          "BB-A8X3-92K4-7QM2",
    "seller_secret": "your-fulfillment_webhook_secret"

Redeem voucher

POST/v1/vouchers/redeemMark the voucher consumed (idempotent).

Call after you've created the paid account on your side. Flips status issued → redeemed and stamps redeemed_at. Re-submitting returns deduped: true without changing state — safe to retry. Refunded / expired / revoked vouchers return 409 with a reason field.

The optional buyer_metadata field (≤2KB JSON) is stored against the voucher for your audit trail.

Seller payouts + ad credit

GET/api/seller-payouts?action=balanceUnsettled + lifetime totals.
GET/api/seller-payouts?action=methodCurrent PayPal payout email.
POST/api/seller-payouts?action=set_methodUpdate PayPal email + display name.
GET/api/seller-payouts?action=historyPaginated dispatched payouts.
POST/api/seller-payouts?action=convert_to_ad_creditConvert MoR earnings to in-platform ad credit.
GET/api/seller-payouts?action=ad_credit_balanceAvailable ad credit + conversion history.

The Promote loop: seller's MoR earnings can be converted into in-platform ad credit (instead of cashing out to PayPal), and that credit can then fund campaigns directly via use_ad_credit: true on campaign create. Money cycles entirely inside BB.

Convert accepts an optional amount field — pass nothing to convert the full unsettled balance, or pass a partial amount which FIFO-walks the captured transactions and rounds up to the nearest one (the response includes a rounded_up flag).

AffAffiliate endpoints

Affiliates save approved products, get a share link, post it anywhere — when someone buys, the affiliate earns 70% of the affiliate pool the seller set. Attribution rides on a bb_click=<uuid> URL parameter that survives Safari ITP and ad blockers.

Share link redirect

GET/s/:token302 to product page with bb_click appended.

Reached via the vercel.json rewrite from /s/:token → /api/affiliate-redirect?token=:token. The handler atomically increments the share link's click counters, fires a best-effort click row into affiliate_clicks, sets the bb_aff_<saved_ad_id> cookie (30-day window — fallback only), and 302-redirects to the product page with ?bb_click=<uuid> appended along with UTM parameters.

This is the entire affiliate-side API for the redirect path — there's no JSON response, just the 302. The bb_click UUID in the URL is the load-bearing attribution identifier, not the cookie.

Sales report

GET/api/affiliate-sales?action=summaryCaptured count, lifetime revenue, lifetime commission, pending settlement.
GET/api/affiliate-salesPaginated sales list (MoR captures attributed to you).

Auth: affiliate JWT. Returns rows from storefront_transactions attributed to this affiliate via bb_click. This is the real money trail — captures BB actually processed on the affiliate's behalf — distinct from the legacy /api/conversions postback flow which records attribution before money has settled.

Benna ranking engine

Benna is the model that ranks every bid served through Boost Boss — both the first-party SuperBoost direct path and the BBX OpenRTB exchange.

!
Honest status. Benna is currently a deterministic targeting-overlap scorer with fixed signal weights, not a learned model. It scores requests against campaign targeting columns (target_intent_tokens, target_active_tools, target_host_apps, target_surfaces) and returns a bid plus an attribution block showing per-signal contribution. The learned ranker is on the roadmap; we'll swap the per-signal weights for a trained table once we have enough outcome data.

Benna powers BBX by default — you don't need to call it directly. The endpoints below exist for transparency (so any auction is inspectable) and for the longer-term plan to license the ranker into other mediation stacks.

Webhooks

There is no unified webhook subscription system today. Three concrete webhook surfaces ship:

Outbound: per-product fulfillment webhook

When a buyer's PayPal capture succeeds, BB fires an HTTPS POST to your product's fulfillment_webhook_url (if set). Use this for non-API server-side provisioning. Payload includes the transaction ID, buyer email, product ID, captured amount, and the voucher code.

Verify the X-BB-Signature header against the fulfillment_webhook_secret you set on the product:

NodeCopy
import crypto from 'crypto';
const sig = req.headers['x-bb-signature'];
const expected = crypto.createHmac('sha256', SECRET).update(req.rawBody).digest('hex');
if (crypto.timingSafeEqual(Buffer.from(sig), Buffer.from(expected))) { // verified }

Inbound: PayPal webhook (every money event)

POST/api/paypal-webhookSingle inbound webhook for the entire money flow.

PayPal is the only payment provider — both rails (advertiser deposits + storefront buyer captures going in, publisher + seller payouts going out) flow through PayPal under the Taiwan-entity single-provider rule. This webhook handles every related event:

  • PAYMENT.CAPTURE.COMPLETED — credits the advertiser balance (deposit) or kicks off the MoR split + voucher issuance (storefront purchase).
  • PAYMENT.CAPTURE.REFUNDED — reverses the corresponding ledger row.
  • PAYMENT.PAYOUTSBATCH.SUCCESS / DENIED — marks publisher and seller payout batches completed or failed.
  • PAYMENT.PAYOUTS-ITEM.SUCCEEDED / FAILED / UNCLAIMED — per-recipient outcome for each payout in a batch.

Verifies the PAYPAL-AUTH-ALGO / PAYPAL-TRANSMISSION-SIG / PAYPAL-TRANSMISSION-ID headers against the configured PayPal webhook ID server-side. Replays are deduped by transmission ID.

i
Legacy Stripe webhook. A /api/stripe-webhook handler still exists for the legacy Stripe pay-in fallback. It's not part of the supported integration surface; expect it to be removed once the Singapore corp move opens up multi-provider options.
i
Roadmap. A unified event-bus model (subscribe to events like ad.served, campaign.approved, voucher.redeemed) is on the longer-term plan but not shipped. Today the dashboard is the canonical surface for these state changes.

Rate limits

The platform applies coarse per-IP rate limits at the Vercel edge and per-account limits inside the handlers. The exact thresholds are tuned for normal operation rather than published as a hard contract — if you're approaching them legitimately, email support@boostboss.ai with the integration profile and we'll raise them. Beacon endpoints (/api/track) are effectively unlimited per impression; auction endpoints are protected to absorb bursts but not unlimited. We'll publish a fixed contract here once production load patterns are stable.

Error codes

Errors return a JSON body with at least an error field (human-readable). Many endpoints also include a code field for programmatic branching:

HTTPCodeMeaning
400invalid_requestMalformed body or missing required field.
400insufficient_creditTried to fund a campaign with use_ad_credit but balance < budget. Response includes available + requested.
401invalid_tokenJWT missing, expired, or malformed.
401bad_secretVoucher API: seller_secret doesn't match the product's fulfillment_webhook_secret.
401no_secretVoucher API: no secret configured on the product.
403forbiddenAuthenticated but not authorized for this resource (e.g. another seller's product).
404voucher_not_foundVoucher code doesn't exist.
404product_missingVoucher references a product that's been deleted.
405—Wrong HTTP method. Body usually names the allowed method.
409refunded / expired / revokedVoucher in a non-redeemable state. The reason field names which.
429rate_limitedToo many requests. Back off and retry with jitter.
500—Server error. Auto-retry safe.
503demo_modeOperation requires Supabase mode (e.g. ad credit funding) but the deployment is in in-memory mode.

Changelog

2026-06-14 — Promote loop (Phase 3)

Campaign create accepts use_ad_credit: true — validates available credit, writes deduction row, stamps credit_funded on the campaign. Closes the MoR earnings → ad credit → campaign funding loop entirely inside BB.

2026-06-13 — Activation link fulfillment is default

Buyer activation now uses one-click links (?bb_token=<code>) into the seller's signup URL by default, replacing the multi-step "paste this code" UX. Voucher API unchanged; only the fulfillment ergonomics improved. Integration guide.

2026-06-11 — Seller payouts + admin dispatch

Sellers configure a PayPal payout email via /api/seller-payouts. Admin dispatch endpoint + Saturday cron settle eligible sellers via PayPal Payouts API. Single-provider rule (PayPal pay-in + payout) per the Taiwan-entity setup.

2026 — Commission Model v2

Replaces the legacy 15% top-line take with a per-product affiliate pool model: seller sets affiliate_pool_pct, BB takes 30% of the pool, affiliate gets 70%, seller keeps the rest. BB only earns when an affiliate drives the sale. Direct (non-affiliate) sales are seller-only minus processor fees.

2026 — MoR storefront live

Buyer-facing product pages at /p/<uuid> with BB-hosted PayPal checkout. Vouchers issued on capture, fulfillment webhook fires to seller. bb_click=<uuid> attribution survives Safari ITP and ad blockers.
