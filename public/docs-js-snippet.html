Lumi SDK script tag

One async <script> tag, then call BoostBoss.requestAd() wherever you want an ad. No backend changes. Works with any LLM provider (OpenAI, Anthropic, Gemini, open-source) and any framework (LangChain, Vercel AI SDK, CrewAI, plain React) — the Web SDK attaches at the rendering surface.

v1.1 — beta. Live on boostboss.ai/sdk.js today.

This page documents the SDK as it actually ships. <script src="https://boostboss.ai/sdk.js"> resolves and works against the live Boost Boss backend. Verify integration end to end with the sandbox key pub_test_demo — no signup required. Demand-side fill is in early beta; expect lower fill rates than mature ad networks until the Founding Publisher cohort fills out.

Prerequisites

  • A Boost Boss publisher account — sign up if you haven't.
  • A web app you control where you can edit the HTML <head> and run a line of JavaScript.
  • Modern browsers (last 2 versions of Chrome, Safari, Firefox, Edge). No IE11.

Installation

Add the snippet to your <head>. It's async — no impact on first paint. The data-api-key attribute auto-initialises the SDK.

<script async
  src="https://boostboss.ai/sdk.js"
  data-api-key="bb_dev_xxx">
</script>

Replace bb_dev_xxx with your publisher key from the dashboard. The global window.BoostBoss is available as soon as the script loads.

Placements

The Web door owns five placements. Each request names a placement and — for the four inline placements — a mount element to render into.

PlacementWhat it isMounts
cornerCorner / sticky anchored unit. Image or video, with a skip countdown.Self-mounts (fixed position)
cardInline sponsored card — headline, optional image, CTA, dismiss button.Into your mount
loadingLoading / "thinking"-state ad — shown while your AI generates a response.Into your mount
citationSponsored source / citation — compact, sits inside a generated answer.Into your mount
chipSponsored suggested-action chip — a tappable quick-reply pill.Into your mount

Requesting an ad

Call BoostBoss.requestAd() with a placement, the request context, and (for inline placements) a mount. It returns a Promise that resolves to the ad object, or null on any error or no-fill.

// Inline sponsored card
await BoostBoss.requestAd({
  placement: 'card',
  context:   'user is comparing CRM tools for a small sales team',
  mount:     '#bb-slot',
});

// Corner unit — self-mounts, no mount needed
BoostBoss.requestAd({ placement: 'corner', context: 'reading a deployment guide' });

Loading-state ads

Request a loading ad when your AI starts generating, then tear it down with clearLoading() when the response is ready.

BoostBoss.requestAd({ placement: 'loading', context: query, mount: '#answer' });
const answer = await generateAnswer(query);
BoostBoss.clearLoading();   // silent — no dismiss event; engagement is the click
renderAnswer(answer);

Context — why it matters

The context string is the single most important argument you pass. It's what the user is currently doing, in plain language. Boost Boss derives a context fingerprint from it: the auction uses it to match relevant demand, and every resulting event — impression, click, skip, dismiss — is logged joined to that context. Vague context yields vague ads; specific context ("user is comparing CRM tools for a small sales team") yields relevant ones. Pass the user's actual message or task, not a page label.

Theming

Pass theme options to init(). The SDK styles itself with CSS variables internally, so it never fights your stylesheet.

BoostBoss.init({
  apiKey:   'bb_dev_xxx',
  theme:    'light',        // 'dark' (default) or 'light'
  accent:   '#6C5CE7',      // CTA / brand accent colour
  position: 'bottom-right', // corner anchor: bottom/top + left/right
});

When you auto-initialise via data-api-key, call init() again afterward to override theme defaults — it merges, it doesn't reset.

Programmatic API

Everything hangs off the global window.BoostBoss.

BoostBoss.init(opts)

Configure the SDK. Optional if you used data-api-key — but required to set theme, callbacks, or caps. Keys: apiKey, theme, accent, position, payToRemovePrice, maxAdsPerSession (default 10), minIntervalMs (default 180000 — applies to corner only), debug, and the callbacks below.

BoostBoss.requestAd(opts)

Request and render an ad. opts.placement (default corner), opts.context, opts.mount (CSS selector or element — required for every placement except corner), optional opts.region / opts.language. Resolves to the ad object or null.

BoostBoss.clearLoading() · BoostBoss.clear(placement)

clearLoading() removes the loading-state ad. clear('card') removes any rendered placement by name. BoostBoss.close() closes the corner / fullscreen popup.

BoostBoss.getLastError() · setDebug(on) · getStats()

getLastError() returns the last { code, message, detail, ts } object or null. setDebug(true) logs every event to the console at runtime. getStats() returns the session id and ads-shown count.

Feedback callbacks

The SDK fires a context-joined tracking beacon for every impression, click, skip, close, and dismiss automatically — you don't wire anything for billing or optimisation to work. The callbacks below are for your own analytics.

BoostBoss.init({
  apiKey: 'bb_dev_xxx',
  onImpression: (ad) => analytics.track('ad_impression', { id: ad.campaign_id }),
  onClick:      (ad) => analytics.track('ad_click',      { id: ad.campaign_id }),
  onClose:      (ad, type) => { /* type: 'close' | 'skip' | 'dismiss' */ },
  onError:      (err) => console.warn(err.code, err.message),
});

Conversion tracking

When a user clicks an ad and converts on the advertiser's own site (a separate thank-you page), the advertiser installs pixel.js on that page — see the conversion beacon docs. Publishers don't wire conversions; you're paid on the impressions and clicks the SDK already reports.

Error handling

The SDK never throws into your app. Every failure resolves requestAd() to null, sets getLastError(), and invokes onError.

Error codeMeaning
NO_API_KEYNo publisher key supplied. Set data-api-key or pass apiKey to init().
BAD_PLACEMENTUnknown placement name. Use one of: corner, card, loading, citation, chip.
NO_MOUNTAn inline placement was requested without a valid mount element.
NO_FILLNo matching advertiser right now. Not a misconfiguration — just empty demand.
RATE_LIMITEDA corner ad was requested inside the minIntervalMs window.
SESSION_CAPmaxAdsPerSession reached for this session.
SERVER_ERROR · FETCH_ERRORHTTP error, network failure, or the 8s request timeout.
EMPTY_RESPONSE · PARSE_ERRORThe auction returned no content or unparseable content.
RENDER_ERRORThe ad was fetched but the placement failed to render into the DOM.

CSP (Content Security Policy)

If your site uses CSP, allow Boost Boss's host:

script-src  'self' https://boostboss.ai;
connect-src 'self' https://boostboss.ai;
img-src     'self' https://boostboss.ai data:;

SSR and SPA support

The SDK is fully client-side — it only runs in the browser, so SSR needs no special handling. On SPA route changes, just call requestAd() again from your route hook (Next.js useRouter, Vue Router afterEach, React Router useLocation). Use clear(placement) to remove a previous ad before re-requesting.

Next.js (App Router)

Add the script in app/layout.tsx via next/script with strategy="afterInteractive", then call BoostBoss.requestAd() from a client component.

import Script from 'next/script';
<Script src="https://boostboss.ai/sdk.js"
  data-api-key="bb_dev_xxx" strategy="afterInteractive" />

Testing

Use the sandbox key pub_test_demo in development. Sandbox returns a fixed creative regardless of context, so you can verify rendering without touching real demand. Add data-debug to log every event to the console.

<script async
  src="https://boostboss.ai/sdk.js"
  data-api-key="pub_test_demo"
  data-debug>
</script>

Switch to your live publisher key and remove data-debug before deploying.

Going live

  • Replace the sandbox key with your live bb_dev_xxx.
  • Remove data-debug.
  • Confirm Stripe Connect is set up for payouts.
  • Add Boost Boss to your CSP if you have one (see above).
  • Verify one impression appears in your dashboard within 60 seconds of going live.

Troubleshooting

requestAd() resolves to null.

Check BoostBoss.getLastError(). NO_FILL means there's no matched advertiser right now; any other code is a misconfiguration (key, mount, CSP, network).

The script blocks page load.

It shouldn't — use the async attribute. If Lighthouse flags a blocking script, confirm async is actually present on the tag.

Theming doesn't apply.

If you auto-initialised via data-api-key, call BoostBoss.init({ theme, accent }) once more after load — the data attribute only sets the key.

Behaviour looks stale after a deploy.

sdk.js is cached aggressively at the CDN. Append ?v=YYYYMMDD to bust it if you suspect a stale SDK.
