Docs/Getting started/Introduction

Boost Boss documentation

Boost Boss is two things at once: a Merchant of Record that handles checkout, fulfillment, and payouts for sellers, and an MCP-native ad network that runs context-aware sponsored content inside AI tools. Four roles share the same account model. Pick the one that matches what you're trying to do.

★ Start hereUpdated Jun 2026·4 min read

Which role are you?

Publisher

You run an AI app, MCP server, or bot

You have an audience using AI tools you build. Drop a Lumi SDK in, return context, get paid 85% of the revenue your impressions generate. Four installation paths — pick the one matching what you ship. Payouts settle biweekly via PayPal.

Get started as a publisher Advertiser

You want to reach AI tool users

Deposit a budget via PayPal, upload your product, pick a goal — Benna handles allocation across all four AI surfaces. Pay only for impressions Benna scored as relevant. Direct boosts through SuperBoost or programmatic via the BBX OpenRTB exchange.

Get started as an advertiser Seller (Merchant of Record)

You sell a product and want BB to handle checkout

BB hosts your product page, takes the payment, splits the proceeds with you and any affiliate, and emails the buyer an activation link that creates the paid account on your service. PayPal pay-in + payout, biweekly settlement.

Get started as a seller Affiliate

You promote products and earn commission

Save any approved product, get a share link, post it on social / newsletter / DM. When someone buys, 70% of the affiliate pool the seller set is yours. Attribution survives Safari ITP and ad blockers via the bb_click= URL parameter. Payouts settle biweekly via PayPal.

Get started as an affiliate

The four roles in one account

You sign in once. The dashboard exposes whatever role(s) you've enabled. Most people start in one role and add another later — a seller who also wants to promote a friend's product becomes an affiliate by saving it; a publisher who wants to advertise their own MCP server becomes an advertiser by depositing budget. The same Supabase user ID carries through, so your payout method, 2FA, and history follow you.

Common concepts

Every role touches the same primitives. Once these click, the surface area of any specific role is small.

Publisher ID
Every publisher account gets an ID like pub_xxx. Use it on every ad request — it's how Boost Boss attributes impressions and revenue.
Context signal
Publishers pass a context string with every ad request — the user's current intent, the tool being called, the page topic, whatever describes "what would be relevant right now." Benna (our ranker) uses it for contextual matching. No cross-site tracking, no behavioral profile.
bb_click attribution
When someone clicks an affiliate share link /s/<token>, BB redirects to the product page with ?bb_click=<UUID> appended. That UUID is what attributes the sale to the affiliate. Survives Safari ITP, ad blockers, and email-link-stripping. Cookie is a secondary fallback only.
Voucher code (activation link)
When a buyer pays for a product, BB issues a single-use BB-XXXX-XXXX-XXXX voucher and emails the buyer a one-click activation link at seller.com/signup?bb_token=<code>. The seller's endpoint validates the code with BB's API and creates the paid account. See Activation link integration.
Commission model v2
The seller sets an affiliate pool as a % of the order. BB takes 30% of that pool, the affiliate gets 70%, the seller keeps the rest of the order. BB only earns when an affiliate drives the sale. Direct (non-affiliate) sales: seller keeps it all minus payment processor fees.
Disclosure
Every ad must be clearly labeled as sponsored. The publisher SDKs handle this automatically — the disclosure label is part of the rendered output. Hiding it is a policy violation.
The Promote loop
Sellers can convert their MoR earnings into in-platform ad credit and use that credit to fund a product boost without ever withdrawing to PayPal. Closes a money cycle entirely inside BB: buyer pays → seller earns → converts to credit → funds boost → reaches more buyers.
Payments — PayPal end to end
Every money rail uses PayPal. Advertiser deposits, buyer storefront captures, publisher payouts, seller payouts, and affiliate payouts all flow through the same provider. This is a deliberate constraint of the Taiwan corporate entity that operates BB today (local law requires pay-in and payout through the same provider). When the Singapore corp move opens up multi-provider options, additional rails will appear on the dashboard's Payout method page — your existing setup carries forward.

Authentication

Boost Boss uses JWT bearer tokens. Sign in via /api/auth?action=login with your email and password; the response includes session.access_token. Pass it on every subsequent request:

Authorization: Bearer <your-jwt>

Tokens expire — re-call login when you get a 401. The token format is a standard signed JWT, not an opaque API key; you can decode it client-side to read your role and user ID without making a network call.

Two special cases use shared secrets instead of user JWTs:

  • Seller voucher API (/v1/vouchers/validate / /v1/vouchers/redeem) — pass seller_secret in the body. The secret is the fulfillment_webhook_secret set on your product.
  • OpenRTB BBX (/api/rtb) — DSPs use a seat key issued at integration time, sent as Authorization: Bearer bb_seat_<key>.

Environments

One base URL: https://boostboss.ai. There's no separate api. subdomain. Two modes:

ModeHow it engagesNotes
Demo / in-memoryDefault when Supabase env vars are not setNo real spend, no real payouts. Useful for SDK integration smoke tests against a self-hosted clone.
ProductionThe live boostboss.ai deploymentReal auctions, real spend, real earnings. The only mode that issues real bb_session_token JWTs and real voucher codes.

API reference

The API reference is the comprehensive list of every shipped endpoint — organized by role. Useful when you want to Ctrl-F something specific. For role-by-role walkthroughs, use the sidebar.

A machine-readable OpenAPI 3.1 spec is also available — drop it into Postman, Stoplight, or your code generator of choice.
